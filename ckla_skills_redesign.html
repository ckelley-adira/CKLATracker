1 System Context & Goals

CKLATracker is the assessment tracking system for Core Knowledge Language Arts (CKLA) at Christel House Indianapolis. It records per-student, per-question scores across unit assessments for Kindergarten through Grade 2 and aggregates those scores into summaries, trend charts, and roster views used by teachers and administrators.

Project Goals

  • Reduce cognitive load and manual navigation for day-to-day score entry
  • Add input validation to eliminate out-of-range and misplaced entries
  • Provide at-a-glance visual feedback (heat maps, sparklines) for teachers and admins
  • Build longitudinal analytics: student progress across all units, skill drill-downs, action summaries
  • Preserve backward compatibility — all existing formulas (AVERAGEIFS, COUNTIFS, quintile buckets) continue to work
  • Lay groundwork for future migration to a dedicated application if needed

2 Current Workflow Audit

The current system is a pair of Google Sheets workbooks (one per grade, with Grade 2 planned). Each workbook contains unit tabs (one per CKLA assessment unit), Roster View tabs (aggregated percentages per student per unit), and Summary Charts tabs (visual trend data).

Teacher Data-Entry Workflow (Manual — Current State)

  1. Open the workbook — Select the correct Google Sheet for the grade level
  2. Find the correct unit tab — Scroll through 10+ tabs to locate the unit being assessed (e.g., "K U8")
  3. Find the student row — Scroll vertically through 105+ student rows to locate the correct student
  4. Find the correct column — Scroll horizontally past metadata columns (A–I), then through up to 132 question columns to find the right one
  5. Enter a score — Type 0 or 1 (or a point value) in the cell
  6. Repeat — Move to the next question column and repeat; then move to the next student row and start over

Estimated time per student: 2–5 minutes (depending on unit size). Per class of 25: 50–125 minutes.

Admin / Reporting Workflow (Current State)

  • Roster View: Navigate to the Roster View tab to see aggregated percentages; no color-coding or heat map is applied
  • Summary Charts: Open Summary Charts tab; Trend column exists but many cells are empty (no SPARKLINE formulas present)
  • Data export: Manual copy-paste into external tools if cross-grade or historical analysis is needed
  • Formula integrity: No cell protection — teachers can accidentally overwrite AVERAGEIFS, COUNTIFS, or quintile formulas

3 Pain Points & Friction Analysis

The following pain points were identified through direct observation, stakeholder interviews, and workbook audit. They are grouped by severity and frequency.

Pain Point Severity Frequency Affected Users
Tab navigation overload — 46+ tabs in a single workbook; no grouping or color-coding High Every session Teachers, Admins
Row/column hunting — Scrolling through 105 rows × 132 columns with no visual anchoring High Every entry Teachers
Blind data entry — No confirmation that the correct cell is targeted; 0/1 values in tiny cells High Every entry Teachers
No input validation — Accepts any value (text, negative numbers, values exceeding max points) Medium Frequent Teachers
Formula cells unprotected — Teachers can accidentally overwrite AVERAGEIFS and quintile formulas High Occasional Admins (fix), Teachers (cause)
No visual status indicators — Roster Views lack heat map formatting; mastery status requires mental calculation Medium Every review Teachers, Admins
Empty Trend columns — Summary Charts have a Trend column but no SPARKLINE formulas are present Low Every review Admins
No audit trail — No record of who entered or changed a score, or when Medium Ongoing Admins
No bulk import — Paper-based assessments must be manually typed one cell at a time Medium After paper tests Teachers
No longitudinal view — No easy way to see a single student's progress across all units Medium Reporting periods Teachers, Admins
Key insight: The highest-friction pain points (tab navigation, row/column hunting, blind entry) are experienced every single time a teacher enters scores. Even small improvements here have an outsized impact on daily workflow.

4 Workbook Architecture

Each grade-level workbook follows a consistent structure. Understanding this layout is critical for both the quick-win improvements and the Apps Script automation layer.

Tab Structure

Tab Type Examples Purpose
Unit Assessment K U1, K U2, … K U10; Gr1 U1, … Gr1 U10 Per-student, per-question score entry. Columns A–I are metadata; J onward are question scores
Roster View Roster View K, Roster View Gr1 Aggregated percentage per student per unit; formula-driven (AVERAGEIFS, COUNTIFS)
Summary Charts Summary Charts K, Summary Charts Gr1 Visual trend data; includes a Trend column (intended for SPARKLINE formulas)
Meta Data Meta Data Student rosters, teacher assignments, grade mappings

Unit Tab Layout (Fixed Row Structure)

Row Content
Row 2 Max points per question (used for validation)
Row 14 Section headers (e.g., "Comprehension", "Vocabulary", "Writing")
Row 15 Question labels (e.g., "Q1", "Q2", … or descriptive names)
Row 16+ Student data rows (one row per student)
Backward compatibility: All improvements are designed to write to the same cells that teachers currently edit manually. Existing AVERAGEIFS, COUNTIFS, quintile formulas, and Summary Charts continue to auto-recalculate with no structural changes.

5 Phase 1 — Quick Wins

Phase 1 These improvements can be applied to the existing workbooks immediately, with no structural rebuild. Estimated effort: 1–2 days.

1. Hide Backend Tabs

Collapse unit assessment tabs behind grade-level groups. Expose only the navigation hub, Roster Views, and Summary Charts to teachers. This reduces the visible tab count from 46+ to under 10.

2. Color-Code Tabs

Apply a consistent color scheme: blue for meta/setup tabs, green for Kindergarten, orange for Grade 1, purple for Grade 2. Teachers can instantly identify which grade they're in.

3. Heat Map Conditional Formatting

Apply green/yellow/red conditional formatting to Roster View percentage columns. Green ≥ 80%, yellow 60–79%, red < 60%. Gives teachers an instant visual read on mastery status without mental calculation.

Implementation: Use HeatMapFormatter.gs from the CKLA Tools → Phase 1 Tools menu.

4. Protect Formula Cells

Lock the AVERAGEIFS, COUNTIFS, and quintile formula columns so only the score-entry columns (J onward) are editable by teachers. Prevents accidental formula overwrites.

5. Data Validation

Restrict score-entry cells to numeric values within the valid range (0 to the max points for each question, as defined in Row 2). Rejects text, negative numbers, and out-of-range values at the cell level.

6. SPARKLINE Trend Formulas

Fill the empty Trend column on Summary Charts tabs with =SPARKLINE() formulas showing each student's unit-by-unit percentage trend.

Implementation: Use SparklineTrend.gs from the CKLA Tools → Phase 1 Tools menu.

7. Checkbox Conversion

Convert 0/1 binary skill columns in unit tabs to Google Sheets checkboxes for faster, less error-prone entry. Checked = 1, unchecked = 0; formulas continue to work with boolean values.

6 Phase 2 — Input Overhaul

Phase 2 Deploy the Google Apps Script system for sidebar-based score entry and automation. Estimated effort: 1–2 weeks. See CKLA_Apps_Script_System.html for full implementation details and code.

Sidebar Score Entry Form

A sidebar form served by Google Apps Script's HtmlService. Teachers select Grade → Unit → Teacher → Student from cascading dropdowns. The form dynamically reads the unit tab's structure (sections from Row 14, questions from Row 15, max points from Row 2) and presents labeled input fields grouped by assessment section, with live validation.

  • Auto-fills existing scores when editing
  • Auto-advances to the next student after submission
  • Estimated entry time: ~30 seconds per student (vs. 2–5 minutes manual)

Per-Teacher Input Sheets

Optional lightweight per-teacher input sheets that sync to the master workbook. Reduces the chance of edit conflicts when multiple teachers enter scores simultaneously.

Navigation Hub

A dedicated hub tab with hyperlinks to all unit tabs, Roster Views, and Summary Charts. Replaces manual tab scrolling with one-click navigation.

Standardized Unit Templates

Template tabs with pre-configured row structure, header rows, formula columns, and data validation. New units can be cloned from the template to ensure consistency.

CSV Import & Backup

Bulk CSV import for paper-based assessment data. Timestamped backup utility creates full copies of grade-level data sheets. Export utility generates CSV files for external analysis.

7 Phase 3 — Reporting & Dashboards

Phase 3 Built-in reporting layer using Apps Script HTML Service — no external BI tools (Looker Studio, etc.) required. Estimated effort: 2–4 weeks.

Student Progress Reports

Historical view of a single student's scores across all units for a grade level. Color-coded by mastery threshold: green ≥ 80%, yellow 60–79%, red < 60%. Printable or exportable as PDF.

Skill Drill-Down by Section

Break down student performance by assessment section (Comprehension, Vocabulary, Writing, etc.) across units. Identify persistent skill gaps that span multiple assessments.

Teacher Action Summaries

Per-teacher reports highlighting students below mastery thresholds, aggregated by class. Designed for data team meetings and intervention planning. Includes count of students at each performance tier.

8 Phase 4 — Scale & Sustainability

Phase 4 Long-term improvements for performance, governance, and potential platform migration. Timeline: future / as needed.

Split Workbooks per Grade

Move each grade level into its own Google Sheets workbook for performance. Large workbooks with 46+ tabs and thousands of formula cells become slow to load and edit. Per-grade splitting also simplifies permissions management.

Audit Trail

Log every score submission (who, when, what changed) to a dedicated "Submission Log" sheet. Enables accountability and error tracing without manual recordkeeping.

Admin Console

Script-driven admin panel for managing tabs (create, hide, protect), refreshing dashboards, running bulk operations, and managing user permissions — all from a single menu interface.

Platform Migration (Evaluate)

If the Google Sheets model becomes unmanageable at scale, evaluate migration to a dedicated application. See Section 9: Platform Options for detailed comparison.

9 Platform Options & Migration Paths

If the spreadsheet-based system reaches its limits (too many students, too many grades, performance issues), the following platforms are viable migration targets. Each is evaluated on ease of migration, feature fit, cost, and maintenance burden.

Platform Strengths Weaknesses Migration Effort
Google Sheets + Apps Script (current) Zero cost; familiar to staff; formulas already built; script layer adds automation Performance degrades at scale; limited UI customization; concurrent editing conflicts N/A (current)
Google AppSheet No-code app on top of Sheets; mobile-friendly forms; role-based views; built-in dashboards Limited formula support; learning curve for admins; may need Workspace license Low — reads existing Sheets data directly
Airtable Relational data model; rich field types; views and filters; automations; API access Cost per seat; data migration required; less familiar to staff; formula differences Medium — requires data restructuring
Custom Web App Full control over UI/UX; relational database; unlimited scalability; tailored reporting Requires development and hosting; ongoing maintenance; highest upfront cost High — full rebuild
Recommendation: Continue with Google Sheets + Apps Script through Phases 1–3. The current system handles the existing student volume and the script layer addresses the core pain points. Re-evaluate platform migration only if student counts exceed ~300 per grade or if multiple schools adopt the system.
